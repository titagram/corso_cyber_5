Opzioni Avanzate delle Regole

Modificatori e PCRE
Rule Buffers

Non dobbiamo cercare l'intero pacchetto per ogni match di contenuto. I Rule Buffers risparmiano tempo e risorse.

Esempio:

alert http any any -> any any (http.accept; content:"image/gif"; sid:1;)

http.accept è uno sticky buffer per match sull'header HTTP Accept. Contiene solo il valore dell'header.

offset e depth

offset: Informa Suricata sulla posizione di inizio all'interno del pacchetto per il matching.

depth: Specifica una lunghezza di byte da considerare per il matching.

Esempio:

content:"|01 02 03|"; offset:0; depth:5;
distance e within

distance: Specifica dove Suricata dovrebbe iniziare a cercare il prossimo pattern di contenuto, misurato dalla fine del match di contenuto precedente.

within: Limita la ricerca a una finestra di ricerca delimitata.

Esempio:

content:"GET"; offset:0; depth:4; content:"/admin"; distance:10; within:80;
dsize

dsize: Match usando la dimensione del payload del pacchetto. Si basa sulla lunghezza del segmento TCP, non sulla lunghezza totale del pacchetto.

Esempio:

alert http any any -> any any (msg:"Large HTTP response"; dsize:>10000; content:"HTTP/1.1 200 OK"; sid:2003;)
PCRE - Perl Compatible Regular Expression

Utilizzare PCRE può essere un game-changer quando si creano regole. Per impiegare PCRE, usiamo l'istruzione pcre seguita da un'espressione regolare.

Nota: Il PCRE dovrebbe essere racchiuso tra barre forward slash iniziali e finali, con eventuali flag posizionati dopo l'ultima barra. Gli anchor sono posizionati dopo e prima delle barre racchiudenti.

Consiglio: Evitare di creare una regola che si basa esclusivamente su PCRE.
